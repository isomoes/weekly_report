# Command Guide

Read this file first. It decides which style guide to use and defines the shared writing rules.

## Step 1: Choose the Writing Mode

Use this routing table:

| User intent | File to read next | Default output style |
| --- | --- | --- |
| weekly report, lab update, progress report, advisor update | `report.md` | objective, technical, paragraph-based |
| tech blog, article, blog intro, blog rewrite, public-facing technical post | `blog.md` | readable, engaging, still technically precise |

If the user asks for both formats from the same source notes, produce them separately and apply each guide independently.

## Step 2: Shared Rules

Apply these rules in every format:

- Preserve technical meaning; do not invent work, metrics, or conclusions.
- Prefer concrete facts over subjective evaluation.
- Keep terminology consistent.
- Remove redundancy and filler.
- Convert fragmented notes into coherent paragraphs unless the user explicitly wants list formatting.
- Add transitions so the text reads as a complete narrative rather than a stitched list.

## Step 3: Format-Specific Adaptation

After reading the chosen guide, adapt the same facts to that format's audience and tone.

- Reports should prioritize traceable actions, measurements, and scope.
- Blogs should prioritize readability, structure, and reader orientation without losing technical accuracy.

## Baseline Failure Patterns to Avoid

Without this skill, writing often drifts into these mistakes:

- weekly reports become vague or mildly promotional
- blogs sound stiff and internal
- strong metrics get reduced to generic claims like "improved performance"
- the same style gets reused for every format
